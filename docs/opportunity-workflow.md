# 机会工作流

## 应用概览

`openSource_scanner` 是一个本地优先的开源机会扫描器，用来把公开的开源项目信号整理成可评估的包装与商业化候选。用户运行扫描命令后，系统从 GitHub 等公开来源收集项目元数据，转换成统一的机会记录，再按许可证、活跃度、使用门槛、包装空间和潜在痛点进行打分。主要移动部分包括数据来源连接器、机会标准化、评分规则、本地 SQLite 历史记录、用户反馈状态和 Markdown 报告。每次运行后的理想结果是一份可读的机会报告：它不承诺收入，只帮助用户更快判断哪些项目值得保存、观察、放弃或进入包装实验。

## 漏斗

1. **Discovery（发现）**：收集公开项目元数据，例如名称、描述、主题、星标、最近更新、许可证和链接。
2. **Scoring（评分）**：用可解释规则评估项目是否有包装价值，重点看痛点强度、活跃度、许可证适配、部署难度和包装信号。
3. **Review（人工复核）**：阅读报告，给候选项目标记 `saved`、`dismissed`、`watch` 或 `package`。
4. **Memo（机会备忘录）**：对 `package` 项写一页简短备忘录，说明目标用户、痛点、包装方式、风险和 7 天验证计划。
5. **Experiment（小实验）**：做一个最小验证，例如托管演示、部署包、行业模板、落地页、中文文档样例或直接访谈。
6. **Feedback（反馈）**：把实验结果和复核动作回写到本地记录中，用来改进后续评分和筛选偏好。

## 合法且有价值的包装路径

- **Hosted SaaS**：把原本需要自部署的工具变成更省心的托管服务，重点提供稳定运行、备份、权限、计费和升级。
- **Deployment Kit（部署套件）**：提供 Docker Compose、Terraform、云平台一键部署、配置模板、监控和运维脚本。
- **Vertical Template（垂直模板）**：把通用开源工具预配置成某个行业或场景的解决方案，例如客服、教育、法律、跨境电商或内部知识库。
- **Localization（本地化）**：补充中文文档、示例数据、教程、术语解释、本地平台适配和中文支持。
- **Integration（集成）**：连接企业已经使用的系统，例如 GitHub、Notion、Slack、飞书、企业微信、数据库、表格或数据仓库。
- **Training / Support（培训与支持）**：提供实施指导、团队培训、故障排查、长期维护和版本升级服务。

## 评估问题

- 目标用户是谁？他们是开发者、运营团队、创业公司、企业部门，还是某个垂直行业角色？
- 这个用户的具体痛点是什么？是部署困难、配置复杂、缺少中文资料、缺少集成、缺少托管，还是缺少可靠支持？
- 为什么用户不直接使用原始开源项目？原项目在哪些地方对目标用户来说太难、太慢或太不确定？
- 付费包装层能让什么事情明显更容易？例如更快上线、更低维护成本、更好集成、更少配置、更清晰教程或更稳定服务。
- 许可证是否适合预期路径？是否允许商业使用、托管、分发、修改和再包装？是否需要保留声明、开源衍生代码或触发额外义务？
- 项目是否仍然活跃？最近是否有提交、issue 处理、release、维护者回应和社区使用迹象？
- 7 天内能否做出可演示的最小成果？如果不能，是否说明包装路径过重或目标用户还不清楚？

## 停止条件

- 许可证带来不可接受的商业、分发或托管风险。
- 原项目长期不活跃、依赖老旧，或维护成本明显高于可获得的价值。
- 目标用户不清楚，无法说清谁会为了什么结果付费。
- 包装方案除了改名、换皮或搬运文档之外，没有增加实质价值。
- 已有强竞争者占据市场，且当前想法没有明确细分人群、渠道、语言、本地化或工作流优势。

## 报告复核动作

- `saved`：项目有价值，但暂时不进入实验；以后可重新评估。
- `dismissed`：项目不适合当前方向；后续报告中应降低优先级或隐藏。
- `watch`：项目值得跟踪，例如增长很快、生态未成熟、许可证或维护状态还需要观察。
- `package`：进入机会备忘录和 7 天小实验，验证是否真的能把开源能力包装成更易用的付费产品、服务或支持方案。

## 使用原则

- 把高分当作“值得进一步看”，不要当作“已经可以商业化”。
- 先验证痛点和可交付价值，再投入完整产品开发。
- 对许可证和商标保持保守判断；有疑问时先停止或寻求专业意见。
- 优先选择能在一周内展示清楚价值的包装路径。
- 每次复核后记录反馈，让扫描器逐步贴近真实偏好，而不是只追逐星标数。
